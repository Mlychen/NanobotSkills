# 风险 05：模块导入依赖 `sys.path` 补丁，复用边界脆弱

## 摘要

当前仓库以“独立 CLI bundle”为主要使用方式，但内部模块导入仍依赖运行时手工修改 `sys.path`。这会让模块复用、测试、嵌入式调用出现脆弱边界。

这类问题对正常 CLI 用户影响较小，但对维护和演进不友好。

## 风险结论

- 风险是否真实：`是`
- 紧急度：`中低`
- 影响面：`低到中` 对 CLI 用户，`中到高` 对开发和复用

## 背景

目前仓库的定位更接近“可复制的自包含脚本 bundle”，而不是一个标准 Python 包。

在这个定位下，CLI 直接运行没有问题。但只要开始出现以下需求，导入组织就会变成问题：

- 测试直接导入内部模块
- 宿主以库形式复用
- 在同一进程里混合不同导入风格

## 证据

### 路径补丁入口

- [../../scripts/timeline_cli.py](../../scripts/timeline_cli.py) 启动时会手工把 `scripts/` 塞进 `sys.path`
- [../../scripts/store.py](../../scripts/store.py) 仍用顶层 `from models import ...`

### 已确认的脆弱场景

- 直接 `from scripts.store import TimelineStore` 会失败，因为 `models` 不是标准包内相对导入。
- 在先导入 `scripts.timeline_cli` 后，再导入 `store` 和 `scripts.store`，会出现双模块身份问题。
- 测试本身已经依赖类似的路径注入技巧。

## 为什么这是风险

### 1. 模块身份可能不一致

同一份代码如果以两个模块名被导入：

- 类型身份会不同
- `isinstance()`、monkeypatch、缓存、调试都可能变脆

### 2. 复用门槛高

当前代码不是不能复用，而是复用者必须先知道：

- 如何补 `sys.path`
- 该导入哪个模块名
- 哪些启动顺序会制造重复模块

### 3. 测试与生产路径差异更大

测试层已经在用路径注入适配内部导入，这会让“测试能跑”和“模块组织健康”变成两回事。

## 影响面

### 影响较小的范围

- 按文档直接运行 `python scripts/timeline_cli.py`
- 按文档通过 `uv run python scripts/timeline_cli.py`

### 影响较大的范围

- 直接导入 `scripts.*` 复用
- 在宿主进程内嵌入
- 测试与工具脚本持续扩展

## 紧急度判断

建议按 `中低` 处理。

当前它更像结构健康问题，而不是用户立刻遇到的主路径故障。

但如果后续明确要把内部模块当库来复用，这一项优先级应立即上调。

## 建议推进方向

### 修复目标

明确代码组织到底是：

- 只支持脚本式 CLI bundle
- 还是也支持包式导入

### 建议步骤

1. 如果只支持 CLI bundle，就尽量减少内部导入暴露，文档明确“不保证包式复用”。
2. 如果支持包式导入，就改成一致的包内导入方式。
3. 测试导入方式与生产推荐方式保持一致。

## 验收建议

至少满足以下之一：

- 方案 A：所有内部模块都能稳定通过包路径导入。
- 方案 B：明确限制只支持 CLI，测试和脚本不再依赖易混淆的双模块导入方式。
