# 风险 06：文档组织分裂，规范源不清晰

## 摘要

当前仓库的设计、合同、使用方式、测试建议分散在多个文档中，且每份文档承担的角色并不完全清晰。

这不会直接打坏运行时，但会提高理解成本、变更成本和合同漂移风险。

## 风险结论

- 风险是否真实：`是`
- 紧急度：`低到中`
- 影响面：`中`

## 背景

这个 skill 的设计本身很强调：

- 公开 contract 稳定
- 内部实现可演进
- 文档与测试同步

在这种前提下，文档应该尽量明确区分：

- 哪份是“规范”
- 哪份是“使用说明”
- 哪份是“设计过程记录”

当前这三种内容混在多个文件里。

## 证据

### 当前主要文档

- [../../SKILL.md](../../SKILL.md)：更偏运行说明、公开命令、测试建议
- [../schema.md](../schema.md)：更偏公开 contract
- [../design-milestones-status-and-acceptance.md](../design-milestones-status-and-acceptance.md)：混合了设计背景、里程碑状态、验收标准、历史决策、性能 profile

### 当前组织问题

- 没有 `README.md` 作为统一入口
- 新接手者很难第一时间判断应该先看哪一份
- 同一件事可能在多个文档里出现，例如 contract、测试入口、设计原则

## 为什么这是风险

### 1. 规范源容易漂移

当同一合同在多份文档中存在时：

- 修改一处，漏改另一处的概率很高
- 后续很难确认“以哪一份为准”

### 2. 接手成本高

新维护者需要先自己建立文档地图，才能进入有效修改。

这会放大：

- 误读设计边界
- 重复讨论已决策问题
- 修复时漏改文档

### 3. 设计文档职责过重

里程碑文档现在同时承担：

- 设计决策
- 实现状态
- 验收标准
- 部分 profiling 记录

这会让它越来越像“工作日志”，而不是易消费的设计资料。

## 影响面

### 直接影响

- 新人 onboarding
- 风险排查
- 变更评审
- 文档同步

### 间接影响

- 会增加 contract 与实现脱节的概率
- 会让维护者更难判断风险优先级和历史边界

## 紧急度判断

建议按 `低到中` 处理。

这项不是当前运行时 bug，但它会持续拖累后续所有变更和修复。

## 建议推进方向

### 修复目标

让文档角色更清晰，降低“同一信息分散在多处”的程度。

### 建议步骤

1. 增加 `README.md` 或在 `references/` 下增加总览入口。
2. 明确：
   - `SKILL.md` 负责什么
   - `schema.md` 负责什么
   - 设计里程碑文档负责什么
3. 对高频引用内容尽量只保留一个规范源，其他文档链接过去。
4. 把历史 profiling / 过程记录与稳定设计结论适度拆开。

## 验收建议

至少做到：

- 新接手者能在 5 分钟内找到“怎么用、合同是什么、设计边界是什么”
- 同一合同只在一处作为规范源维护
- 其他文档主要用链接引用，而不是重复复制
